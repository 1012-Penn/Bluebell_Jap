## 🏗️ 项目架构概述

### 什么是项目架构？
项目架构是指软件项目的整体结构和组织方式，包括：
- **目录结构**：文件和文件夹的组织方式
- **代码分层**：不同功能模块的职责划分
- **依赖关系**：各个模块之间的调用关系
- **配置管理**：环境配置和参数管理
- **部署方式**：如何将项目部署到生产环境

### 为什么需要好的项目架构？
- **可维护性**：代码结构清晰，易于理解和修改
- **可扩展性**：新功能可以轻松集成到现有架构中
- **团队协作**：多人开发时，职责分工明确
- **代码复用**：通用功能可以独立成模块，重复使用
- **测试友好**：各层独立，便于单元测试

### 目录命名规范
- **小写字母**：所有目录名使用小写字母
- **下划线分隔**：多个单词用下划线连接（如：init_sql）
- **语义化命名**：目录名要能清楚表达其用途
- **避免缩写**：除非是通用缩写（如：dao、dao等）

## 🏛️ 分层架构设计

### 经典三层架构
```
┌─────────────────────────────────────┐
│           表现层 (Presentation)      │  ← controller/
│         HTTP请求处理、路由           │
├─────────────────────────────────────┤
│           业务层 (Business)          │  ← logic/
│         业务逻辑、数据处理           │
├─────────────────────────────────────┤
│           数据层 (Data)              │  ← dao/, models/
│        数据库操作、数据模型          │
└─────────────────────────────────────┘
```

### 各层职责详解

#### 1. 表现层 (controller/)
```go
// 职责：处理HTTP请求，参数验证，调用业务层
func (c *UserController) CreateUser(ctx *gin.Context) {
    // 1. 参数绑定和验证
    var req CreateUserRequest
    if err := ctx.ShouldBindJSON(&req); err != nil {
        response.Error(ctx, err)
        return
    }
    
    // 2. 调用业务层
    user, err := c.userService.Create(req)
    if err != nil {
        response.Error(ctx, err)
        return
    }
    
    // 3. 返回响应
    response.Success(ctx, user)
}
```

#### 2. 业务层 (logic/)
```go
// 职责：业务逻辑处理，数据验证，调用数据层
func (s *UserService) Create(req CreateUserRequest) (*User, error) {
    // 1. 业务规则验证
    if err := s.validateUser(req); err != nil {
        return nil, err
    }
    
    // 2. 数据转换
    user := &User{
        Name:  req.Name,
        Email: req.Email,
    }
    
    // 3. 调用数据层
    return s.userRepo.Create(user)
}
```

#### 3. 数据层 (dao/, models/)
```go
// 职责：数据库操作，数据模型定义
type User struct {
    ID       int    `json:"id" db:"id"`
    Name     string `json:"name" db:"name"`
    Email    string `json:"email" db:"email"`
}

func (r *UserRepository) Create(user *User) (*User, error) {
    // 数据库插入操作
    result := r.db.Create(user)
    if result.Error != nil {
        return nil, result.Error
    }
    return user, nil
}
```

### 依赖注入原则
- **上层依赖下层**：controller → service → repository
- **避免循环依赖**：各层之间不能相互依赖
- **接口解耦**：通过接口定义依赖关系
- **依赖注入**：通过构造函数注入依赖

## 📦 依赖管理

### Go模块系统
```go
// go.mod 文件
module github.com/your-username/bluebell

go 1.21

require (
    github.com/gin-gonic/gin v1.9.1
    github.com/go-sql-driver/mysql v1.7.1
    // 其他依赖...
)
```

### 依赖分类
1. **核心依赖**：项目运行必需的包
2. **开发依赖**：开发、测试时使用的包
3. **间接依赖**：被其他包依赖的包

### 依赖管理最佳实践
- **版本锁定**：使用 `go mod tidy` 锁定版本
- **定期更新**：定期更新依赖到最新稳定版本
- **安全扫描**：使用 `govulncheck` 检查安全漏洞
- **依赖审查**：审查新添加的依赖

## ⚙️ 配置文件管理

### 配置目录结构
```
conf/
├── config.go        # 配置结构体定义
├── config.yaml      # 配置文件
├── config.dev.yaml  # 开发环境配置
├── config.prod.yaml # 生产环境配置
└── config_test.yaml # 测试环境配置
```

### 配置管理原则
1. **环境分离**：不同环境使用不同配置文件
2. **敏感信息**：密码、密钥等不提交到代码库
3. **配置验证**：启动时验证配置的完整性
4. **热重载**：支持运行时重新加载配置

### 配置示例
```yaml
# config.yaml
server:
  port: 8080
  mode: debug

database:
  host: localhost
  port: 3306
  username: root
  password: ${DB_PASSWORD}
  database: bluebell

redis:
  host: localhost
  port: 6379
  password: ""
  db: 0
```

## 🚀 部署配置

### Docker配置
```dockerfile
# Dockerfile
FROM golang:1.21-alpine AS builder

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download

COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -a -installsuffix cgo -o main .

FROM alpine:latest
RUN apk --no-cache add ca-certificates
WORKDIR /root/

COPY --from=builder /app/main .
COPY --from=builder /app/conf ./conf

EXPOSE 8080
CMD ["./main"]
```

### Docker Compose配置
```yaml
# docker-compose.yml
version: '3.8'

services:
  app:
    build: .
    ports:
      - "8080:8080"
    depends_on:
      - mysql
      - redis
    environment:
      - DB_HOST=mysql
      - REDIS_HOST=redis
    volumes:
      - ./conf:/root/conf

  mysql:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: password
      MYSQL_DATABASE: bluebell
    ports:
      - "3306:3306"
    volumes:
      - mysql_data:/var/lib/mysql

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  mysql_data:
```

## 💡 最佳实践

### 1. 代码组织原则
- **单一职责**：每个文件、函数只负责一件事
- **开闭原则**：对扩展开放，对修改关闭
- **依赖倒置**：依赖抽象，不依赖具体实现
- **接口隔离**：接口要小而专一

### 2. 错误处理
```go
// 统一错误处理
type AppError struct {
    Code    int    `json:"code"`
    Message string `json:"message"`
    Details string `json:"details,omitempty"`
}

func HandleError(c *gin.Context, err error) {
    var appErr *AppError
    if errors.As(err, &appErr) {
        c.JSON(appErr.Code, appErr)
        return
    }
    
    // 默认错误处理
    c.JSON(500, AppError{
        Code:    500,
        Message: "服务器内部错误",
    })
}
```

### 3. 日志管理
```go
// 结构化日志
logger.Info("用户创建成功",
    "user_id", user.ID,
    "username", user.Name,
    "timestamp", time.Now(),
)
```

### 4. 测试策略
- **单元测试**：测试各个函数的功能
- **集成测试**：测试模块间的协作
- **端到端测试**：测试完整的业务流程

### 5. 性能优化
- **连接池**：数据库、Redis连接池管理
- **缓存策略**：合理使用缓存减少数据库查询
- **异步处理**：耗时操作异步处理
- **资源监控**：监控内存、CPU使用情况

## 🎯 总结

好的项目架构是成功开发的基础。通过合理的目录结构、清晰的分层设计、完善的配置管理和现代化的开发工具，我们可以：

1. **提高开发效率**：清晰的架构让开发更高效
2. **降低维护成本**：结构化的代码易于维护
3. **提升代码质量**：分层架构促进代码复用
4. **支持团队协作**：明确的职责分工便于协作
5. **便于项目扩展**：良好的架构支持功能扩展

记住：**架构不是一成不变的，要根据项目需求和技术发展不断演进**。
